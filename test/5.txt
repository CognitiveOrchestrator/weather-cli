## 核心代码改进建议及优化方案

### 1. 城市解析模块日志污染问题

**问题**：
`resolveCity.ts` 模块内部直接使用 `console.log` 输出"解析成功"信息，当启用 `--json` 模式时，这些日志会混入 JSON 输出，破坏数据格式的纯净性，导致下游解析工具失败。

**建议**：
移除内部硬编码的日志输出，改为返回结构化解析结果对象（如 `{ resolvedName: string, source: 'baidu'|'input' }`），由主调程序根据 `options.json` 标志决定是否打印辅助信息。

**实现示例**：

```typescript
export async function resolveCity(city: string): Promise<{ resolvedName: string; fromApi: boolean }> {
  // ... 调用百度API
  if (results && results.length > 0) {
    return { resolvedName: results[0].cityName, fromApi: true };
  }
  return { resolvedName: city, fromApi: false };
}
```

**主程序中调用方式**：

```typescript
const { resolvedName, fromApi } = await resolveCity(inputCity);
if (!options.json && fromApi) {
  console.log(chalk.green(`✅ 城市解析成功: ${inputCity} → ${resolvedName}`));
}
```

### 2. 国内城市空气质量数据准确性

**问题**：
国内城市 AQI 数据使用硬编码的北京坐标（116.3975, 39.9087），导致所有中国城市空气质量数据完全失真，无法反映真实污染状况。

**建议**：
1. 通过高德地理编码 API 实时获取城市精确坐标
2. 或建立本地 `cityAdcodes` 到经纬度的映射表
3. 使用获取的坐标调用 Open-Meteo 空气质量 API
4. 如不愿增加额外请求，应明确说明当前仅国际城市支持准确 AQI

**实现示例**：

```typescript
async function getCityCoordinates(city: string): Promise<{ lat: number; lon: number } | null> {
  // 使用高德地理编码API或本地映射（例如从cityAdcodes查经纬度表）
  // 若没有，返回null
}

// 在fetchWeather后，如果需要AQI
if (options.aqi) {
  const coords = await getCityCoordinates(targetCity);
  if (coords) {
    weatherData.aqi = await fetchAirQuality(coords.lat, coords.lon) || undefined;
  } else {
    console.warn(chalk.yellow('无法获取该城市坐标，空气质量数据不可用'));
  }
}
```

### 3. 城市映射表完善与标准化

**问题**：
- **拼写错误**：`'tiefa'` 应为 `'tieling'`（铁岭）；`'taizhou_zj'` 可能重复（台州已有 `'taizhou'`）
- **中文键缺失**：国际城市映射缺少 `'香港'`、`'澳门'` 等中文键，用户输入中文时无法匹配
- **标准化不足**：输入处理未移除空格或进行大小写统一，降低匹配成功率

**建议**：
1. 彻底审查并修正所有城市映射条目
2. 补充国际城市的中文键名
3. 对用户输入进行标准化预处理（移除空格、转小写）
4. 建立城市别名映射，支持常见变体

**实现示例**：

```typescript
const internationalCities: Record<string, { lat: number; lon: number; name: string }> = {
  // 添加中文键
  '香港': { lat: 22.3193, lon: 114.1694, name: '香港' },
  '澳门': { lat: 22.1987, lon: 113.5439, name: '澳门' },
  // 确保所有英文键无空格变体
  'hongkong': { lat: 22.3193, lon: 114.1694, name: '香港' },
};

// 标准化输入处理
const normalized = city.replace(/\s+/g, '').toLowerCase();
return internationalCities[normalized] || null;
```

### 4. 高德API类型定义缺失

**问题**：
高德气象 API 响应处理使用大量 `as any` 类型断言，完全绕过 TypeScript 类型检查，失去静态类型安全保障，增加运行时错误风险。

**建议**：
定义完整的高德 API 响应类型接口，包括：
- `GaodeLiveResponse`（实时天气响应）
- `GaodeForecastResponse`（预报响应） 
- `GaodeLocation`（地理位置响应）
- `GaodeCityInfo`（城市信息）

**关键改进点**：
- 消除所有 `as any` 类型断言
- 为嵌套对象定义详细接口
- 添加字段可选性标注（哪些字段可能缺失）
- 统一定义响应错误类型

### 5. Fetch超时控制优化

**问题**：
两个独立的 fetch 请求（实时天气和多日预报）共用一个 AbortController，当一个请求超时，另一个也同时被中止，Promise.all 中任一失败整体失败的逻辑虽可接受，但不够精细，可能造成不必要的数据丢失。

**建议**：
1. 为每个 fetch 单独设置超时控制器
2. 或使用 `Promise.race` 包装单个 API 调用，实现独立超时控制
3. 考虑使用 `Promise.allSettled` 替代 `Promise.all`，确保部分成功时仍能获取可用数据

**实现策略**：
- 高德 API 和 Open-Meteo API 设置不同超时阈值（国内 API 可能更快）
- 实现请求重试机制（针对临时网络故障）
- 超时后提供缓存备用数据（如有）

### 6. 壁纸模式宽度计算简化

**问题**：
壁纸模式使用 `string-width` 计算内容宽度并手动居中，而 `boxen` 库原生支持 `alignment` 选项，当前实现存在不必要的手动计算复杂性。

**建议**：
1. 直接使用 `boxen` 的 `alignment: 'center'` 选项
2. 或保持手动居中，但确保边框字符宽度与内容宽度严格一致
3. 统一宽度计算逻辑，消除硬编码的魔法数字

**优化效果**：
- 代码更简洁，维护成本降低
- 对齐效果更稳定，减少终端差异影响
- 支持动态宽度调整（根据终端尺寸自适应）

### 7. 缓存键设计优化

**问题**：
缓存键为 `weather_${city.toLowerCase()}`，但同一城市可能有多种输入形式（如"北京"、"beijing"、"BEIJING"、"Bei Jing"），导致缓存无法共享，频繁重复请求同一城市数据。

**建议**：
1. **方案A**：使用统一城市标识符，国内城市用 adcode，国际城市用 `lat,lon` 字符串
2. **方案B**：标准化城市输入，建立城市别名映射，统一转换为规范名称
3. **方案C**：接受当前限制，但缩短缓存时间（如15分钟），影响有限

**推荐方案**：
```typescript
// 国内城市：省份编码+城市编码（如110100）
// 国际城市：经纬度字符串（如"22.3193,114.1694"）
const cacheKey = isChineseCity ? `weather_${adcode}` : `weather_${lat},${lon}`;
```

### 8. 高德API风力转换优化

**问题**：
`parseWindPower` 函数将风力等级简单乘以5近似转换为 km/h，转换公式粗糙，与实际风速对应关系不准确，可能误导用户。

**建议**：
1. **方案A**：保留风力等级显示（如"3级"），更符合中文用户习惯
2. **方案B**：使用更精确的转换公式，不同风力等级对应不同转换系数
3. **方案C**：同时显示风力和风速，提供完整信息

**风力等级对应表**：
| 等级 | 风速范围(km/h) | 描述 |
|------|----------------|------|
| 1级 | 1-5 | 软风 |
| 2级 | 6-11 | 轻风 |
| 3级 | 12-19 | 微风 |
| 4级 | 20-28 | 和风 |
| 5级 | 29-38 | 清劲风 |

### 9. 错误消息本地化

**问题**：
部分错误消息为英文（如"Failed to fetch"、"Network error"、"City not found"），对中文用户不友好，尤其不利于非技术用户理解问题原因。

**建议**：
1. 统一所有错误消息为中文
2. 根据错误类型提供具体解决方案提示
3. 实现错误分级：信息性提示、警告、致命错误
4. 支持错误消息模板，便于后续国际化扩展

**错误消息示例**：
- ✅ "网络请求失败，请检查网络连接"
- ✅ "城市'${city}'未找到，请检查拼写或尝试附近城市"
- ✅ "API密钥无效或过期，请更新.env文件"
- ✅ "服务暂时不可用，请稍后重试"

### 10. 单元测试框架缺失

**问题**：
核心功能函数（如 `formatWeather`、`parseWindPower`、`convertTemp` 等）缺乏单元测试，代码改动后无法快速验证功能正确性，依赖手动测试，效率低下且易出错。

**建议**：
1. 为所有纯函数添加单元测试
2. 使用 Jest 测试框架 + nock 模拟 HTTP 请求
3. 建立测试数据工厂，生成各种边界条件测试用例
4. 集成到 CI/CD 流程，确保每次提交自动运行测试

**测试覆盖重点**：
- 格式化函数：测试不同输入组合的输出格式
- 转换函数：测试边界值、异常输入处理
- API适配器：测试网络错误、超时、无效响应
- 缓存逻辑：测试过期策略、键冲突处理
